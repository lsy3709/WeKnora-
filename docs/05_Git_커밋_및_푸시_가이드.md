# 05. Git 가이드: 커밋 및 푸시 (Commit & Push)

이 문서는 실습 과정에서 수정된 코드를 안전하게 저장소에 반영하기 위한 Git 사용 방법을 안내합니다.

## 1. 변경 사항 확인 (`status`)
작업을 마친 후 어떤 파일이 수정되었는지 확인합니다.
```bash
git status
```

## 2. 변경 사항 스테이징 (`add`)
커밋할 파일을 선택합니다. 모든 변경 사항을 추가하려면 `.`을 사용합니다.
```bash
# 특정 파일만 추가
git add docs/00_설치_가이드.md

# 모든 변경 사항 추가
git add .
```

## 3. 커밋 메시지 작성 (`commit`)
변경 내용을 요약하여 기록합니다. 교육 과정에서는 다음과 같은 **커밋 컨벤션** 사용을 권장합니다.

### 📝 커밋 메시지 규칙 (Convention)
- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정 (Markdown 등)
- `style`: 코드 포맷 변경 (세미콜론 누락 등)
- `refactor`: 코드 리팩토링
- `test`: 테스트 코드 추가

**사용 예시:**
```bash
git commit -m "docs: 설치 가이드 한글화 및 튜토리얼 추가"
git commit -m "feat: 데이터 분석 차트 생성 기능 구현"
```

## 4. 저장소에 반영 (`push`)
내 로컬 컴퓨터의 변경 사항을 원격 저장소(GitHub)로 업로드합니다.
```bash
git push origin main
```

---

## ⚠️ 주의 사항 및 팁

### 1. 푸시 전 최신 코드 가져오기 (`pull`)
다른 사람이 코드를 수정했을 수 있으므로, 푸시하기 전에 항상 최신 코드를 먼저 가져오는 습관이 중요합니다.
```bash
git pull origin main
```

### 2. .env 파일 주의
보안을 위해 **`.env` 파일은 절대 커밋하지 마세요.** 
이미 `.gitignore`에 등록되어 있지만, 중요한 API 키가 유출되지 않도록 항상 주의해야 합니다.

### 3. 커밋 단위는 작게
너무 많은 변경 사항을 한꺼번에 커밋하기보다, 논리적인 작업 단위로 나누어 커밋하는 것이 나중에 문제를 찾거나 협업할 때 훨씬 유리합니다.
